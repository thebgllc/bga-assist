# BGA i18n / Translation — do it up front

Wire translation in as you build, not as a pre-submission retrofit. BGA's
`project-check` flags every player-facing English literal it thinks isn't
translatable ("Possibly untranslated 'X'"), and going back to fix a finished
game touches dozens of lines across PHP data, PHP notifications, and JS. The
rules below are cheap if applied while writing each string and are derived from
a real cleanup (Salvage Divers).

---

## The one-sentence model

A string is translatable **only if its exact literal was collected into the
translation catalog** — and BGA collects the catalog by *textually scanning
source* for `clienttranslate('literal')` (PHP) and `_('literal')` (JS). At
runtime, `clienttranslate()`/`_()` just return the source string; the actual
per-language swap happens client-side against that catalog. Two consequences:

1. **A literal must appear inside `clienttranslate(...)` or `_(...)` somewhere**
   for translators to ever see it. A bare `'Reinforced Fins'` in a data array
   is invisible to the extractor → untranslatable.
2. **`_(variable)` translates at runtime but declares nothing.** The extractor
   can't see the value behind a variable, so the literal must *also* be
   declared as a `clienttranslate('...')` literal elsewhere (an "anchor") or it
   won't be in the catalog. This is exactly what `project-check`'s
   "variable content will be translated only if it matches a string declared
   with clienttranslate()" info message means.

## Where each kind of string goes

| String lives in… | Do this |
|---|---|
| A PHP notification message | `clienttranslate('You drew ${chip}')` as the message arg (already standard). |
| A PHP state `description` / `descriptionMyTurn` | Wrap in `clienttranslate('...')` (the scaffold does this). |
| A **substituted arg** inside a notif message (`${name}`) | Add `'i18n' => ['name']` to the notif args so the *value* is translated too, and make sure the value's literal is anchored (below). |
| A **JS literal** shown in the UI | `_('Draw a chip')` — the JS extractor sees it directly. |
| A **JS `_(variable)`** (e.g. `_(site.name)`, `_(item.name)`) | Wrap the display in `_()` **and** anchor every possible literal value with `clienttranslate()` in a PHP file (see anchors). |
| **Player-facing names in a data catalog** (chips, cards, market items, tiles) | Wrap the name literal in `clienttranslate(...)` at the point of definition. If that file is framework-free, use the Core no-op shim (below). |

## Anchors — for strings the extractor can't see

When the client renders a name via `_(item.name)` / `_(site.name)`, add a
never-called PHP method whose body lists each literal. It runs never; it exists
only to be scanned:

```php
/** i18n anchors: strings the client translates via _(variable). Never called —
 *  the body exists purely for BGA's textual string collection. Keep in sync. */
private function i18nClientAnchors(): array
{
    return [
        clienttranslate('The Kaiser Wreck'),
        clienttranslate('The Sunken Cathedral'),
        // …one per possible runtime value…
    ];
}
```

## Framework-free Core data (the shim)

If your player-facing names live in a **framework-independent** layer (a pure
PHP engine reused by a simulator/tests that has no BGA `\clienttranslate`),
you can't call the global function without breaking the non-BGA build. Define a
**namespaced no-op** so the extractor still collects the literals while the Core
stays standalone:

```php
namespace Bga\Games\YourGame\Core;

// Player-facing catalog names are wrapped in clienttranslate() so BGA's textual
// extractor collects them; this no-op keeps the Core sim/tests framework-free.
// In the BGA runtime an unqualified clienttranslate() in this namespace resolves
// HERE (not to global \clienttranslate) and returns the source string unchanged —
// which is exactly what clienttranslate does anyway; translation is client-side.
if (!function_exists(__NAMESPACE__ . '\\clienttranslate')) {
    function clienttranslate(string $text): string { return $text; }
}
```

Then wrap the names in place: `new MarketItem('fins1', clienttranslate('Reinforced Fins 1'), …)`.
Put the shim in a file that always loads before the names are used (e.g. the top
of the catalog class file, after the `namespace` line). The extractor matches on
the text `clienttranslate(` regardless of namespace, so the strings are still
collected; PHP function-name fallback means BGA's own runtime is unaffected.

## Do NOT translate

- **IDs / keys** (`'fins1'`, `'white-1'`, `COLOR_GEAR`) — never shown; wrapping
  them pollutes the catalog and can collide.
- **Dev-only / sim-only strings** — bot names, policy labels, debug/log text
  that never reaches a player. Wrapping them dumps meaningless entries on the
  volunteer translators. Leave them raw and accept the `project-check` warning
  (or, better, keep dev-only files off the Studio deploy entirely).

## Checklist when adding any new player-facing string

1. Is it a JS literal? → `_('...')`. Done.
2. Is it a PHP message/description? → `clienttranslate('...')`. Done.
3. Is it a data-catalog name shown to players? → wrap the name literal in
   `clienttranslate(...)` (Core shim if framework-free) **and** render it client-
   side with `_(name)`.
4. Is it substituted into a notif message (`${x}`)? → add `'i18n' => ['x']`.
5. Is it a `_(variable)` on the client? → add a `clienttranslate('...')` anchor
   for every possible value.
6. Is it dev-only / never player-visible? → leave it raw.

English-first launch is fine (BGA translation is community-driven post-launch) —
but doing the above as you write costs nothing and keeps `project-check` clean.
